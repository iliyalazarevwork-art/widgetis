# One Plus One — налаштування та сценарії акції

## Як працює акція

Правило: на кожні **N товарів** (за замовчуванням 3) → найдешевший товар, у якого є клон, замінюється на клон за 1 грн.

Кількість подарунків = `floor(total / min_items)`.

### Архітектура

1. **Фронтенд** (widget на сайті магазину) слухає події кошика Horoshop
2. При зміні кошика — відправляє дані на бекенд (`/api/one-plus-one/evaluate`)
3. **Бекенд** рахує кількість подарунків і повертає swap-інструкції
4. Фронтенд виконує операції в кошику Horoshop (add/reduce/remove) в 3 фази

### Фази виконання операцій

Horoshop скидає qty оригіналу при зміні клону, тому операції виконуються послідовно:

1. **Phase 1**: `appendProduct` — додавання нових клонів
2. **Phase 2**: `setQuantity` — зміна qty існуючих клонів (збільшення/зменшення)
3. **Phase 3**: `reduce/remove` — зменшення/видалення оригіналів

Між фазами — settle (800мс тиші).

## Сценарії

### Один тип товару (А = 360 грн, має клон)

| Кошик | Реальна к-ть | Подарунків | Результат | Коментар |
|---|---|---|---|---|
| А×1 | 1 | 0 | А×1 | менше 3 |
| А×2 | 2 | 0 | А×2 | менше 3 |
| А×3 | 3 | 1 | А×2 + клон×1 | найдешевший = А |
| А×4 | 4 | 1 | А×3 + клон×1 | |
| А×5 | 5 | 1 | А×4 + клон×1 | |
| А×6 | 6 | 2 | А×4 + клон×2 | |
| А×9 | 9 | 3 | А×6 + клон×3 | |

### Два різних товари, обидва мають клони (А=360, Б=490)

| Кошик | Реальна к-ть | Подарунків | Результат | Коментар |
|---|---|---|---|---|
| А×1 + Б×1 | 2 | 0 | без змін | менше 3 |
| А×1 + Б×2 | 3 | 1 | Б×2 + клонА×1 | А дешевший → він за 1 грн |
| А×2 + Б×1 | 3 | 1 | Б×1 + А×1 + клонА×1 | А дешевший |
| А×2 + Б×2 | 4 | 1 | Б×2 + А×1 + клонА×1 | А дешевший |
| А×3 + Б×3 | 6 | 2 | Б×3 + А×1 + клонА×2 | 2 найдешевших = 2×А |

### Два товари, тільки А має клон

| Кошик | Реальна к-ть | Подарунків | Результат | Коментар |
|---|---|---|---|---|
| А×1 + Б×2 | 3 | 1 | Б×2 + клонА×1 | А має клон → він за 1 грн |
| А×1 + Б×3 | 4 | 1 | Б×3 + клонА×1 | |

### Два товари, А=360 (з клоном), Б=200 (дешевше, але БЕЗ клону)

| Кошик | Реальна к-ть | Подарунків | Результат | Коментар |
|---|---|---|---|---|
| А×1 + Б×2 | 3 | 1 | Б×2 + клонА×1 | Б дешевший, але **немає клону** → беремо А |
| А×2 + Б×4 | 6 | 2 | Б×4 + клонА×2 | |

### Зміна найдешевшого (додаємо дешевший товар з клоном)

| До | Дія | Реальна к-ть | Подарунків | Результат |
|---|---|---|---|---|
| Б×2 + клонА×1 | додати В (200 грн, з клоном) | 4 | 1 | Б×2 + А×1 + клонВ×1 (В дешевший → він за 1 грн, А повертається) |
| Б×2 + А×1 + клонВ×1 | видалити В | 3 | 1 | Б×2 + клонА×1 (В немає → знов А) |

### Зменшення кількості

| До | Дія | Реальна к-ть | Подарунків | Результат |
|---|---|---|---|---|
| А×4 + клон×2 (6 шт) | зменшити А (клік −) | 5 | 1 | А×4 + клон×1 |
| А×2 + клон×1 (3 шт) | зменшити А (клік −) | 2 | 0 | А×2 (клон видалено, оригінал відновлено) |

### Видалення клону вручну (opt-out)

| До | Дія | Результат |
|---|---|---|
| Б×2 + клонА×1 (3 шт) | видалити клон | Б×2 + А×0 = 2 шт, акція не діє (артикул А в opt-out) |
| Б×2 (після opt-out) | додати новий товар | opt-out скидається, evaluate перераховує |
| Б×2 (після opt-out) | видалити Б | opt-out скидається |
| А×4 + клон×2 (6 шт) | видалити клон | А×4, акція не свопає А (opt-out), але якщо є інші товари з клонами — свопає їх |

### Edge cases

| Ситуація | Що відбувається |
|---|---|
| Видалення клону вручну | Артикул оригіналу додається в opt-out, клон НЕ повертається |
| opt-out скидається | При додаванні або видаленні не-клонного товару |
| Зменшення qty нижче min_items | Клони видаляються, оригінали відновлюються |
| Швидке натискання +++ | Debounce 600 мс, один evaluate |
| Порожній кошик | Все чиститься |
| Оригінал повністю замінений (qty→0) | Трекається в SwapRecord, включається в cartData при повторному evaluate |
| Horoshop стріляє зайві events | Settle-based suppress (800 мс тиші) |
| Кнопки +/- на клонах | Заблоковані (opacity 0.4, pointer-events: none) |

## Налаштування нового магазину

### 1. Завантаження каталогу на R2

Каталог — xlsx-файл з колонками: `Артикул`, `Назва`, `Розділ`, `Ціна`.
Клони товарів мають артикул з суфіксом `-1uah` (наприклад: `ABC123` → `ABC123-1uah`).

```bash
task cdn-upload -- ./path/to/catalog.xlsx catalogs/shop-name.xlsx
```

Публічний URL після завантаження:
```
https://pub-fec3d8c4f8f34ee29c041d1a4ec7080d.r2.dev/catalogs/shop-name.xlsx
```

### 2. Створення промо в базі даних

```bash
ssh root@204.168.206.10
docker exec horoshop-widget-market-postgres-1 psql -U postgres -d widgetality
```

```sql
INSERT INTO one_plus_one_promos
  (name, site, is_active, catalog_url, catalog_format, categories, product_map, article_suffix, min_items, one_uah_price, settings, created_at, updated_at)
VALUES
  (
    'Назва акції',           -- довільна назва
    'shop.example.com',      -- домен магазину (без https://)
    true,                    -- активна
    'https://pub-fec3d8c4f8f34ee29c041d1a4ec7080d.r2.dev/catalogs/example.xlsx',
    'xlsx',                  -- формат: xlsx | csv | json
    NULL,                    -- NULL = всі категорії, або JSON-масив: '["Чоловікам/Худі"]'
    '{}',
    '-1uah',                 -- суфікс артикула клона
    3,                       -- мінімум товарів для спрацювання акції
    1.00,                    -- ціна клона (грн)
    NULL,
    NOW(), NOW()
  )
ON CONFLICT (site) DO UPDATE SET
  catalog_url = EXCLUDED.catalog_url,
  is_active   = EXCLUDED.is_active,
  updated_at  = NOW();
```

### 3. Побудова product_map

Після створення промо — побудувати маппінг з каталогу та зберегти в БД:

```bash
task prod-artisan -- one-plus-one:build-map shop.example.com
```

Або для всіх сайтів:

```bash
task prod-artisan -- one-plus-one:build-map
```

Маппінг зберігається в колонку `product_map` таблиці `one_plus_one_promos`.
Evaluate тепер бере маппінг з БД — не парсить xlsx при кожному запиті.

### 4. Додавання Horoshop ID товарів

```bash
task prod-artisan -- scrape:product-ids shop.example.com /path/to/catalog.xlsx --limit=0 --delay=500
```

Або вручну:

```sql
INSERT INTO horoshop_products (site, article, horoshop_id, created_at, updated_at)
VALUES ('shop.example.com', 'ABC123', 12345, NOW(), NOW())
ON CONFLICT (site, article) DO UPDATE SET horoshop_id = EXCLUDED.horoshop_id, updated_at = NOW();
```

### 5. Оновлення каталогу

Після зміни xlsx на R2:

```bash
task prod-artisan -- one-plus-one:build-map shop.example.com
```

Це оновить `product_map` в БД і очистить кеш Redis.
